# הנחיות לפיתוח: העברת משחק שחמט לארכיטקטורת שרת-לקוח (WebSockets)

## 📌 סקירת הפרויקט ומצב נוכחי
ברשותנו לוגיקת משחק עובדת וממשק גרפי (GUI) של לוח המשחק ברמת הלוקאל. המטרה הנוכחית היא להעביר את האפליקציה למודל שרת-לקוח מבוסס תהליך בודד (Single-process) באמצעות תקשורת WebSockets, תוך יישום הדרגתי של ניהול משתמשים, חדרים ומערכת שידוכים.

---

## 🏗️ ארכיטקטורה כללית
1. **השרת (Server):** מנהל את מצב המשחק הרשמי (Source of Truth), מאמת מהלכים, ומנהל את החיבורים והחדרים.
2. **הלקוח (Client):** מציג את לוח המשחק, קולט קלטים מהמשתמש (הקלקות/גרירות), שולח פקודות לשרת, ומעדכן את הממשק הגרפי לפי העדכונים המתקבלים מהשרת.

---

## 🛠️ שלב 1: תשתית התקשורת הבסיסית (Pub/Sub & WebSockets)
### 1. מימוש אפיק הודעות (Pub/Sub Bus) בצד השרת
יש לממש או להשתמש ברכיב Bus פנימי בשרת שיאפשר רישום (Subscribe) והפצה (Publish) של אירועים.
* **אירועים נדרשים באפיק:**
  * `update_scores` - עדכון תוצאות משחק.
  * `update_move_logs` - רישום מהלכים שבוצעו.
  * `play_sound` - טריגר להפעלת אפקטים קוליים אצל הלקוחות.
  * `game_animation` - הפעלת אנימציות פתיחה או סיום משחק.

### 2. פרוטוקול הודעות (WebSocket Protocol)
כל ההודעות בין הלקוח לשרת יועברו בפורמט JSON במבנה הבא:
```json
{
  "type": "שם_האירוע",
  "payload": { ... נתוני ההודעה ... }
}
```

#### ⬅️ מהלקוח לשרת (Commands):
* **שליחת מהלך:** `{"type": "make_move", "payload": {"move": "WQe2e5"}}` (דוגמה: המלכה הלבנה נעה מ-e2 ל-e5).
* **בקשת הצטרפות לחדר:** `{"type": "join_room", "payload": {"room_id": "1234"}}`

#### ➡️ מהשרת ללקוח (State Updates):
* **עדכון מצב לוח:** `{"type": "game_state", "payload": {"board": "...", "turn": "white"}}`
* **הפעלת צליל/אנימציה:** `{"type": "trigger_event", "payload": {"event": "move_sound"}}`

## 👥 שלב 2: ניהול משתמשים ודירוג (SQLite & Shell Login)
1. **התחברות (Login):**
   * אין לפתח ממשק גרפי (GUI) למסך החיבור. תהליך הרישום וההתחברות יתבצע דרך ה-Shell/Terminal של הלקוח בעת הפעלתו.
   * השרת ישמור את המשתמשים (שם משתמש + סיסמה מוצפנת) במסד נתונים מקומי מסוג **SQLite**.
2. **מערכת דירוג (ELO Rating):**
   * כל משתמש חדש מתחיל עם דירוג של `1200`.
   * בסיום כל משחק, השרת יחשב מחדש את ה-ELO של שני השחקנים ויעדכן את מסד הנתונים.

---

## ⚔️ שלב 3: מנגנון שידוכים (Matchmaking) וטיפול בניתוקים
1. **לחצן "Play" במסך הבית:**
   * בעת לחיצה, הלקוח שולח בקשה לשרת להיכנס לתור השידוכים.
   * השרת יחפש שחקן נוסף שנמצא בתור ודירוג ה-ELO שלו הוא בטווח של `+-100` מהשחקן הנוכחי.
   * **טיימאאוט:** אם לא נמצא שחקן תוך 60 שניות, השרת ישלח הודעת שגיאה והלקוח יציג הודעה קופצת (Popup) שלא נמצא משחק.
2. **טיפול בניתוקים (Disconnect Grace Period):**
   * אם שחקן מתנתק באמצע משחק פעיל, השרת לא מסיים את המשחק מייד אלא מתחיל ספירה לאחור של **20 שניות**.
   * הלקוח של השחקן שנותר מחובר יציג טיימר ספירה לאחור על המסך.
   * אם השחקן המנותק לא חוזר תוך 20 שניות, נקבע לו הפסד טכני (Auto-resign).

---

## 🚪 שלב 4: מערכת חדרים (Rooms) וצפייה (Spectating)
1. **לחצן "Room" במסך הבית:**
   * יפתח חלונית קופצת עם שדה טקסט ו-3 כפתורים: **Create**, **Join**, **Cancel**.
   * **Create:** השרת מייצר מזהה ייחודי (Room ID) ומחזיר אותו ללקוח. המזהה יוצג בחלק העליון של המסך.
   * **Join:** השחקן מזין מזהה חדר ונכנס אליו.
2. **חלוקת תפקידים בתוך חדר:**
   * המשתמש ה-1 שמצטרף/יוצר = שחקן לבן (White).
   * המשתמש ה-2 שמצטרף = שחקן שחור (Black) -> המשחק מתחיל באופן אוטומטי.
   * המשתמש ה-3 ואילך שמצטרפים = צופים (Viewers). צופים מקבלים את עדכוני ה-`game_state` אך השרת יחסום כל פקודת `make_move` מצידם.

---

## 📝 שלב 5: ניטור ויומנים (Logging)
* יש לממש מערכת לוגים מקיפה (הן בצד השרת והן בצד הלקוח).
* כל פעולה (התחברות, שליחת מהלך, החלפת מצב, ניתוק, הודעת שגיאה) חייבת להירשם בקובץ לוג ייעודי עם חותמת זמן (Timestamp) לצורך דיבאגינג קל.

---

## 🚀 שלבי עבודה מומלצים לסוכן:
1. **צעד ראשון:** העברת לוגיקת המשחק הקיימת לתוך מחלקת מנהל משחק (`GameManager`) בצד השרת, ויצירת חיבור WebSocket בסיסי שמסנכרן את הלוח בין שני דפדפנים/חלונות.
2. **צעד שני:** בניית ה-Bus הפנימי בשרת והצמדת הצלילים והאנימציות לאירועים שלו.
3. **צעד שלישי:** הקמת מסד הנתונים SQLite ותהליך ההתחברות מה-Shell.
4. **צעד רביעי:** פיתוח לוגיקת החדרים (Rooms) והשידוכים (Matchmaking).
5. **צעד חמישי:** הוספת מנגנון הניתוקים והטיימר.
